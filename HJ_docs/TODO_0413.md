### 2.1 SQP Planner (default `dynamic_avoidance_mode=SQP`)

**경로**: `planner/sqp_planner/`

**노드:**
- `dynamic_sqp_server.py` — dynamic_reconfigure 서버
- `sqp_avoidance_node.py` — 메인 (591 lines)
- `update_waypoints.py` — 속도 업데이트 (122 lines)

**입력:**
- `/tracking/obstacles`, `/opponent_prediction/obstacles`
- `/car_state/odom_frenet`, `/car_state/odom`
- `/global_waypoints[_scaled]`, `/global_waypoints_updated`
- `/behavior_strategy`, `/opponent_trajectory`, `/ot_section_check`
- `/planner/avoidance/smart_static_active` (HJ)

**출력:**
- `/planner/avoidance/otwpnts` (OTWpntArray)
- `/planner/avoidance/markers_sqp` (MarkerArray)
- `/planner/avoidance/merger` (Float32MultiArray)

**알고리즘 핵심:**
1. 장애물 필터링 (lateral threshold + lookahead)
2. **ROC** (Region of Consideration) 그룹핑 — 겹치는 장애물 묶음
3. **SQP 최적화** (SLSQP, max 20 iter):
   - Objective: `Σd² + Σd''² + (d'[0])²`
   - 제약: 시작/끝 라인 정렬, heading, 장애물 회피, 곡률, side consistency

**3D 적합성:**
| 항목 | 상태 |
|------|------|
| FrenetConverter 3D 초기화 | ✅ 이미 z 사용 |
| Obstacle 메시지 | ❌ 2D (z_m 없음) |
| 곡률 계산 (xy 기반) | ✅ controller가 xy heading만 사용 |
| GridFilter 의존성 | ✅ 없음 |
| Track3DValidator 필요? | ⚠️ 현재 d_left/d_right만으로 체크 → 3D 트랙 검증 추가 권장 |

**하드코딩 / 이슈:**
- Line 63: `min_radius = 0.55m` (wheelbase)
- Line 65: `car_width = 0.30m`
- Line 85: CCMA `w_ma=10, w_cc=3`
- Line 465: `μ = 0.318` (그립)
- Line 345: dead `print("this happend")` TODO
- Line 425: `d[-2]` constraint (off-by-one 의심)
- Line 481: side_consistency가 array 반환 (이상한 constraint format)
- Line 365-366: 결과 정렬이 trajectory 순서 깰 수 있음
- 곡률 한계 계산: `μ * g / v²` (`g = 9.81`)

## 5. 공통 고려사항

### 5.1 Obstacle 메시지 — z 추가 검토
현재 `Obstacle.msg`:
```
int32 id
float64 s_start, s_end, s_center
float64 d_center, d_left, d_right
float64 size, vs, vd
bool is_static, is_visible
```
**z 좌표 없음.** 3D 트랙에서 장애물이 정말 의미 있으려면 z_m 추가 검토.

→ 단기: perception이 z를 안 주니 무시하고 진행
→ 장기: perception 3D화 시 동시에 추가

### 5.2 Track3DValidator 활용
- recovery/static에서 한 것처럼, **dynamic에도 boundary raycast 적용** 검토
- SQP는 d_left/d_right만 쓰니 추가 검증 가치 큼
- spliner_planner는 GridFilter 대체로 필수

### 5.3 리샘플링 일관성
- recovery/static에 적용한 등간격 리샘플링 패턴 그대로
- s,d 일관성 (single source `get_frenet`)
- d_m 보정 (`ref.d_right + d_m`, `ref.d_left - d_m`)

### 5.4 state machine 변경 필요한가?
- 거의 없음. `update_velocity()`가 어떤 플래너 wpnt든 동일하게 처리
