### 1.3 핵심 문제: 궤적 유효성 검증 (Collision Checking)

**2D 방식 (현재):**
```
GridFilter(map_topic="/map") → OccupancyGrid → PNG erode → is_point_inside(x,y)
```
- `grid_filter.py:127-138`: world→pixel 변환 후 eroded_image[py,px] == 255 체크
- 3D 트랙에서는 `/map` 토픽 자체가 의미 없음

**3D 대안 분석:**

| 방식 | 장점 | 단점 | 적합성 |
|------|------|------|--------|
| **A. d_right/d_left 범위 체크** | 이미 데이터 있음, 빠름 | 수평면 투영, 뱅킹 시 부정확 | △ 1단계 필터 |
| **B. FrenetConverter boundary raycast** | 3D wall geometry 사용, 이미 구현됨 | per-point (경계근처만) | ✅ **2단계 정밀 검증** |
| **C. 3D Voxel/SDF** | 정밀 | 구현 부담 大, 메모리 | ❌ 과잉 |
| **D. PCD ray casting** | 실제 벽 PCD 활용 | 런타임 벽 PCD 필요 | ❌ 런타임 부적합 |
| **E. 3D Mesh** | 최고 정밀 | 메쉬 생성+point-in-mesh 무거움 | ❌ 과잉 |

**채택: A+B 하이브리드** — d_right/d_left로 빠른 필터 + **s-based** boundary raycast로 정밀 검증
- boundary를 init 시 s로 매핑 → runtime에 s 기반 로컬 필터 → **XY 겹침 문제 완전 해결**
- z-filter 대신 s-filter 사용: 같은 XY라도 s가 다르면 다른 구간으로 정확히 구분

## 2. 핵심 설계: 3D 궤적 유효성 검증 (GridFilter 대체)

### 2.1 채택 방식: **3단계 하이브리드 — d-bound 필터 + boundary raycast (80Hz 대응)**

**성능 요구: 80Hz = 12.5ms budget per iteration**

**핵심 제약: `get_frenet_3d()`는 Python for-loop → 100pt에 ~30ms → 사용 불가**
- `get_frenet()` (2D, 벡터화 numpy)은 100pt에 ~1.5ms → OK
- 이 트랙은 오버랩 구간 없음 → 2D Frenet으로 s,d 충분
- z는 `spline_z(s)`로 별도 보간

**3단계 검증 로직 (GridFilter 대체):**
```python
class Track3DValidator:
    """GridFilter 대체 — 3단계 하이브리드, 80Hz 대응"""
    
    def __init__(self, converter, gb_wpnts, safety_margin=0.15):
        self.converter = converter
        self.wpnt_dist = gb_wpnts[1].s_m - gb_wpnts[0].s_m
        self.max_idx = len(gb_wpnts)
        self.safety_margin = safety_margin
        # d_right/d_left 배열 프리컴퓨트 (벡터 연산용)
        self.d_left_arr = np.array([w.d_left for w in gb_wpnts])
        self.d_right_arr = np.array([w.d_right for w in gb_wpnts])
    
    def validate_trajectory(self, samples_xy, s_arr):
        """
        전체 궤적 유효성 검증 (벡터화)
        Returns: (valid: bool, first_invalid_idx: int or -1)
        """
        # === Stage 1: d_right/d_left 범위 체크 (벡터화, ~0.1ms) ===
        _, d_arr = self.converter.get_frenet(samples_xy[:, 0], samples_xy[:, 1])
        wpnt_idxs = (s_arr / self.wpnt_dist).astype(int) % self.max_idx
        
        d_left = self.d_left_arr[wpnt_idxs]
        d_right = self.d_right_arr[wpnt_idxs]
        
        outside = (d_arr > d_left - self.safety_margin) | \
                  (d_arr < d_right + self.safety_margin)
        
        if not np.any(outside):
            return True, -1  # 전부 안전 → 바로 통과
        
        # === Stage 2: 배치 벡터 wall crossing — s-based 필터 (~0.3ms) ===
        # 전체 궤적 세그먼트 × 벽 세그먼트를 numpy 배치 연산
        # s 기반 로컬 필터 → XY 겹침 구간에서도 정확
        # Python for loop 없음, 전부 벡터 연산
        crossing = self.check_trajectory_wall_crossing(
            samples_xy[:-1], samples_xy[1:], s_arr[:-1]
        )
        if np.any(crossing):
            return False, int(np.argmax(crossing))
        
        # Stage 1에서 outside였지만 벽은 안 뚫음 → 경계 매우 근접
        first_outside = int(np.argmax(outside))
        return False, first_outside
```

**성능 예산 (100 points, 80Hz = 12.5ms):**
```
Spline BPoly 생성:              ~0.5ms
get_frenet (2D, 벡터화):        ~1.5ms
z 보간 spline_z(s):             ~0.2ms
Stage 1 d-bound (벡터화):       ~0.1ms
Stage 2 배치 ray casting:       ~0.3ms  (s-based, (N,M) 배치 벡터, for loop 0개)
Heading/curvature 계산:          ~0.3ms
Wpnt 조립 + Marker publish:     ~0.5ms
─────────────────────────────────
총합:                            ~3.4ms ✅ (12.5ms 이내)
```

**Init 시 1회 전처리:**
- boundary → s 매핑: centerline과 1:1 대응 (trackbound markers/2 == centerline waypoints)
- s_margin, wrap-around 프리컴퓨트

**왜 이게 d_right/d_left 단독보다 나은가:**
- Stage 2가 **실제 3D 벽 geometry** (`/trackbounds/markers`)를 사용

- 궤적의 **경로 자체**가 벽을 뚫는지 체크 (점 하나하나가 아님)
- FrenetConverter에 이미 구현돼 있으므로 추가 작업 최소화

**`_build_track_3d_params()` 이식** (`global_velocity_planner_3d.py:286-334`):
```python
def _build_track_3d_params(self, wpnts):
    mu = np.array([wp.mu_rad for wp in wpnts])     # ← 플래너가 채워야 함!
    kappa = np.array([wp.kappa_radpm for wp in wpnts])
    s = np.array([wp.s_m for wp in wpnts])          # ← 플래너가 채워야 함!
    
    if np.all(np.abs(mu) < 1e-8):
        return mu, None  # 평면 구간 → 2D 동작
    
    slope = mu
    dmu_ds = np.gradient(slope, s)
    phi = np.zeros(len(wpnts))
    
    # Euler→body Jacobian으로 angular rates 계산
    omega_x = -np.sin(mu) * kappa
    omega_y = np.cos(phi) * dmu_ds + np.cos(mu) * np.sin(phi) * kappa
    omega_z = -np.sin(phi) * dmu_ds + np.cos(mu) * np.cos(phi) * kappa
    
    return slope, {
        'mu': mu, 'phi': phi,
        'omega_x': omega_x, 'omega_y': omega_y, 'omega_z': omega_z,
        'd_omega_x': np.gradient(omega_x, s),
        'd_omega_y': np.gradient(omega_y, s),
        'd_omega_z': np.gradient(omega_z, s),
        'dmu_ds': dmu_ds,
        'h': self.h_cog,
        'slope_correction': self.slope_correction,
        'slope_brake_margin': self.slope_brake_margin,
        'slope_brake_vmax': self.slope_brake_vmax,
    }
```

**따라서 플래너가 반드시 채워야 하는 필드:**
- `mu_rad` — slope angle (GB wpnt에서 복사)
- `s_m` — arc length (Frenet 변환 결과)
- `kappa_radpm` — curvature (스플라인에서 계산)

**추가 rosparam 로드 (3D state machine `__init__`):**

- `slope_correction`, `slope_brake_margin`, `slope_brake_vmax`: vel planner yaml

**friction sector**: 2.5D vel planner 내부에서 `/friction_map_params` rosparam 자동 조회 → 별도 전달 불필요

#### 4.2.3 Spline 생성: 2D → 3D 확장
**원본 (`static_avoidance_node.py:328-376`):**
- `get_cartesian(s, d)` → (x, y) 만 반환
- `BPoly.from_derivatives` → 2D (x, y)
- tangent = `[cos(psi), sin(psi)]` (2D)

**3D 버전:**
- `get_cartesian_3d(s, d)` → (x, y, z) 반환
  - 주의: `get_cartesian_3d`는 z를 spline_z(s)에서 가져옴 → d 방향 오프셋은 xy 평면에서만 적용
  - **이는 충분함**: 레이싱 트랙에서 lateral offset은 수평면 기준이고, z는 트랙 표면 높이를 따라감
- spline은 **xy만 BPoly** + z는 레퍼런스 경로에서 보간
  - 이유: 스플라인으로 z를 독립 피팅하면 트랙 표면에서 이탈 가능
  - recovery_spliner도 동일 패턴 (xy만 스플라인, z는 참조 경로)
- tangent은 여전히 2D `[cos(psi), sin(psi)]` 유지
  - 스플라인 접선은 수평면 heading이므로 2D가 맞음

**구현 패턴:**
```python
# Frenet → 3D Cartesian
resp_3d = self.converter.get_cartesian_3d(s_array, d_array)  # (3, N)
# control points (xy만)
points.append(resp_3d[:2, i])  # x, y
z_ref = resp_3d[2, :]          # z는 별도 보관

# BPoly 스플라인 (xy)
samples_xy = ...  # 기존과 동일

# z 보간: 각 sample의 s 좌표 → spline_z(s)
s_samples, _ = self.converter.get_frenet(samples_xy[:, 0], samples_xy[:, 1])
z_samples = self.converter.spline_z(s_samples)
samples_xyz = np.column_stack([samples_xy, z_samples])
```

#### 4.2.4 웨이포인트 필드 완전 채우기 (핵심!)

**문제**: 현재 `xyv_to_wpnts()`가 채우는 필드:
- ✅ id, x_m, y_m, s_m, d_m, vx_mps, psi_rad, kappa_radpm
- ❌ **z_m** → 기본값 0.0 → controller가 z=0으로 추적 → 3D에서 치명적
- ❌ **mu_rad** → 기본값 0.0 → slope 정보 없음
- ❌ **d_right, d_left** → 기본값 0.0 → controller safety_ratio = 0
- ❌ **ax_mps2** → state machine `update_velocity()`에서 채움 (OK)

**Controller가 사용하는 필드** (`controller_manager.py:544-555`):
```python
[x_m, y_m, z_m, speed, safety_ratio, s_m, kappa_radpm, psi_rad, ax_mps2, d_m]
#                ^^^                  ^^^^^^^^^^^^^^^^
#              z_m 필수!            d_right+d_left로 계산!
```

**3D `xyv_to_wpnts()` 수정:**
```python
def xyv_to_wpnts(self, x, y, z, s, d, v, psi, kappa, wpnts, gb_wpnt_idx):
    wpnt = Wpnt()
    wpnt.id = len(wpnts.wpnts)
    wpnt.x_m = x
    wpnt.y_m = y
    wpnt.z_m = z                                          # ← 3D 추가
    wpnt.s_m = s
    wpnt.d_m = d
    wpnt.vx_mps = v
    wpnt.psi_rad = psi
    wpnt.kappa_radpm = kappa
    # GB 웨이포인트에서 가져오는 필드들
    ref_wpnt = self.gb_wpnts[gb_wpnt_idx]
    wpnt.d_right = ref_wpnt.d_right                       # ← 트랙 경계
    wpnt.d_left = ref_wpnt.d_left                         # ← 트랙 경계
    wpnt.mu_rad = ref_wpnt.mu_rad                         # ← 3D slope
    # ax_mps2는 state machine update_velocity()에서 재계산됨
    return wpnt
```

**설계 근거:**
- `z_m`: 스플라인에서 계산한 실제 3D z좌표 사용
- `d_right/d_left`: 해당 s 위치의 GB 웨이포인트에서 복사 (스플라인 포인트가 트랙 어디에 있든 해당 구간의 트랙 폭은 동일)
- `mu_rad`: 해당 s 위치의 GB 웨이포인트에서 복사 (slope은 위치 의존적)
- `vx_mps`: **state machine이 `update_velocity()`로 재계산**하므로 플래너 값은 참고용

#### 4.2.5 속도: state machine에서 재계산

**State machine `update_velocity()` (`state_machine_node.py:1382-1431`):**
- 플래너가 넣은 `vx_mps`는 **덮어쓰임**
- `calc_vel_profile(kappa, el_lengths, ggv, v_start, v_end, ...)` 기반 재계산
- `v_end`는 `gb_wpnts[...].vx_mps`에서 가져옴 → 3D velocity profile 반영됨
- `ax_mps2`도 `calc_ax_profile()`으로 재계산

**따라서 플래너에서 속도를 정교하게 넣을 필요 없음** — state machine이 처리
- 단, `kappa_radpm`은 정확해야 함 (속도 프로파일 입력)
- `el_lengths`도 정확해야 함 (x,y 기반 거리 계산)

**rqt 3D velocity 편집 반영 경로:**
```
rqt → global_velocity_planner_3d → /global_waypoints (vx_mps 재계산)
    → vel_scaler_node → /global_waypoints_scaled (sector scaling)
    → state_machine gb_wpnts → update_velocity()의 v_end 참조
    → local_waypoints의 최종 vx_mps에 반영됨
```

#### 4.2.6 Frenet 변환: 2D 벡터화 유지 (80Hz 대응)

## 6. 속도 파이프라인 확인 사항

### 6.1 속도 흐름 (최종 정리)
```
GB Optimizer (global_waypoints.json)
    ↓ [/global_waypoints: base vx_mps]
Global Velocity Planner 3D (global_velocity_planner_3d.py)
    ↓ [/global_waypoints: 3D slope/grip 보정 vx_mps] ← rqt 편집 반영
Vel Scaler (vel_scaler_node.py)
    ↓ [/global_waypoints_scaled: sector별 스케일링 vx_mps] ← rqt 섹터 편집 반영
State Machine (gb_wpnts에 저장)
    ↓
플래너 (recovery/static) → 경로 생성 (x,y,z,s,d,psi,kappa + z_m,mu_rad,d_right,d_left)
    ↓ [/planner/recovery/wpnts, /planner/avoidance/static_otwpnts]
State Machine update_velocity() — **2.5D vel_planner_25d 사용**
    ↓ vx_mps 재계산 (kappa + slope + track_3d_params + grip_scale_exp + friction)
    ↓ v_end=gb_wpnts_scaled[s].vx_mps (rqt 속도 반영)
    ↓ ax_mps2 재계산 (calc_ax_profile)
    ↓ [local_waypoints: 모든 필드 완전한 Wpnt]
Controller
```

### 6.2 핵심: 속도는 state machine이 2.5D vel planner로 입힘
- **플래너가 넣는 vx_mps는 무시됨** — state machine `update_velocity()`에서 재계산
- 3D state machine은 `vel_planner_25d` 사용 → slope, track_3d_params, grip_scale_exp 전달
- `v_end`는 `gb_wpnts[glb_start_idx].vx_mps` → 이미 3D velocity profile + sector scaling 반영
- **플래너가 정확히 채워야 하는 입력들:**
  - `kappa_radpm` — 속도 프로파일 계산의 핵심 입력
  - `mu_rad` — slope 보정 입력 (3D 고유)
  - `s_m` — track_3d_params 빌드 시 gradient 계산에 사용
  - `x_m, y_m` — el_lengths 계산에 사용
- `ax_mps2`도 state machine이 `calc_ax_profile()`로 재계산
- **friction sector**: 2.5D vel planner가 내부에서 rosparam 자동 조회

### 6.3 rqt 편집 반영 경로
```
사용자 rqt 편집 (v_max, ax_max 등)
    → global_velocity_planner_3d: vx_mps 재계산, /global_waypoints 재발행
    → vel_scaler_node: sector scaling 적용, /global_waypoints_scaled 재발행
    → state_machine: gb_wpnts 갱신
    → update_velocity(): v_end 참조 시 갱신된 속도 사용
    → 다음 local_waypoints 발행 시 반영됨
```

### 6.4 플래너에서 해야 할 것 vs 안 해도 되는 것
| 항목 | 플래너 책임 | State Machine 책임 | 2.5D vel planner 사용 여부 |
|------|-----------|-------------------|--------------------------|
| x_m, y_m, z_m | ✅ 정확히 채움 | ❌ 안 건드림 | el_lengths 계산 (x,y) |
| **s_m** | ✅ **정확히 채움 (필수)** | ❌ 안 건드림 | **track_3d_params gradient 계산** |
| d_m | ✅ 정확히 채움 | ❌ 안 건드림 | — |
| **kappa_radpm** | ✅ **정확히 채움 (필수)** | ❌ 안 건드림 | **속도 프로파일 핵심 입력** |
| psi_rad | ✅ 정확히 채움 | ❌ 안 건드림 | — |
| d_right, d_left | ✅ GB wpnt에서 복사 | ❌ 안 건드림 | — |

## 10. 리스크 및 판단 보류 사항

### 9.1 Track3DValidator 정확성
- d_right/d_left는 글로벌 웨이포인트 간격(~0.1m)마다 정의됨
- 웨이포인트 사이 보간이 필요할 수 있음 (선형 보간으로 충분할 것)
- **보수적 safety_margin (0.15~0.20m)으로 커버 가능**

### 9.2 Frenet 변환 성능
- `get_frenet_3d()`는 per-point로 height filter + boundary raycast 수행
- 스플라인 포인트 100~200개 × 20Hz = 초당 2000~4000회 호출
- **Python 벡터화 구현이므로 실시간 가능 여부 프로파일링 필요**
- 대안: `get_frenet()` (2D) 사용하되, z는 별도 spline_z(s) 보간
  - 3D 트랙 오버랩이 없는 구간에서는 2D로 충분

### 9.3 장애물 좌표계
- `/tracking/obstacles`의 obstacle이 3D Frenet (s,d)을 정확히 제공하는지 확인 필요
- perception 모듈이 아직 3D에서 비활성 → **실제 장애물 입력 없이 테스트 시 수동 Publish**

### 9.4 State Machine 호환성
- State machine은 이미 3D 지원 (IY/HJ 수정 완료)
- `SmartStaticChecker` 관련 로직은 smart 미사용 시 bypass됨
- Recovery 전환 조건 (`_check_on_spline`, `_check_free_frenet`)은 Frenet 기반 → 3D에서도 유효

### 9.5 Dynamic Avoidance (향후)
- 현재 스코프 밖: prediction, opponent tracking 필요
- static avoidance가 안정되면 dynamic으로 확장 예정
- dynamic의 핵심 차이: 예측 궤적 기반 충돌 검사 → Track3DValidator 재활용 가능
