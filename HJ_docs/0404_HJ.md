## 2. 분석: 2.5D vs 3D NLP 속도 프로파일 비교

### 2-1. 테스트 트랙: hj_test_0403

| 항목 | 2.5D (mincurv + FB) | 3D NLP (TUMRT time-optimal) |
|---|---|---|
| 옵티마이저 | mincurv QP | CasADi NLP |
| 경로 전략 | 곡률 최소화 | 시간 최소화 (경로+속도 커플링) |
| max |kappa| | 1.06 rad/m | 5.50 rad/m |
| 속도 범위 | 2.1 ~ 8.1 m/s | 2.7 ~ 8.6 m/s |
| 평균 속도 | 4.4 m/s | 6.0 m/s |
| 랩타임 | 21.5s | 15.5s |
| slope 범위 | -6.4 ~ 8.2 deg | -6.7 ~ 8.5 deg |
| z 범위 | 0.138 ~ 0.600 m | 0.137 ~ 0.601 m |

### 2-2. 프로파일이 다른 이유

속도 프로파일 차이는 **3D 고려 부족이 아님**.
**근본적으로 다른 경로 최적화 방식** 때문:

- **mincurv**: 부드러운 라인, 낮은 최대 곡률 → 코너에서 일관된 속도
- **mintime NLP**: 코너를 타이트하게 자름 (kappa 5.5까지) → 직선에서 빠르지만 코너에서 급감속

같은 트랙, 다른 경로 → 다른 속도 프로파일. 이건 정상.

### 2-3. TUMRT 3D 대비 2.5D에서 빠진 것

**이미 적용 중:**
- `grip_scale = cos(slope)` — 경사면에서 수직 하중 감소
- `ax_gravity = -g * sin(slope)` — 종방향 중력 외력

**아직 미적용 (TUMRT는 이것들을 함):**
- **V_omega 항**: `V² * (dmu/ds)` — 볼록/오목 지형에서 수직 하중 변화
  - 볼록 (언덕 꼭대기, dmu/ds < 0): 그립 감소, 차가 뜰 수 있음
  - 오목 (골짜기 바닥, dmu/ds > 0): 그립 증가
- **경사면 코너링**: `V² * kappa * sin(slope)` — slope 위에서 코너링할 때 수직 하중 변화
- **g_tilde 클램프**: `max(g_tilde, 0)` — 음수 수직하중 방지 (차가 뜨는 것 방지)

15컬럼 CSV에 필요한 데이터 다 있음: mu_rad, dmu_radpm, omega_y_radpm

## 3. 계획: 속도 프로파일 고도화

### 3-1. g_tilde 고도화 (calc_ax_poss)

**현재** (vel_planner_25d/vel_planner.py):
```python
grip_scale = math.cos(slope)
ax_gravity = -9.81 * math.sin(slope)
```

**제안:**
```python
g_tilde = (g * cos(slope)
           + vx_start**2 * dmu_ds              # 볼록/오목 지형 효과
           + vx_start**2 * (1.0/radius) * sin(slope))  # 경사면 코너링
g_tilde = max(g_tilde, 0.0)                    # 차 뜨는 것 방지
grip_scale = g_tilde / g
ax_gravity = -g * sin(slope)
```

**TUMRT의 g_tilde 전체 식 (참고):**
```
g_tilde = fmax(w_dot - V_omega + (omega_x² - omega_y²) * h + g*cos(mu)*cos(phi), 0.0)
```
- w_dot: 횡이동 가속도의 수직 성분 (뱅크 없으면 ≈0, 적용 불가)
- V_omega: 속도×도로회전 상호작용 = `V² * dmu/ds` (적용 가능)
- centrifugal: 원심력 수직 성분, h=0.074m이라 영향 작음 (옵션)
- g*cos(mu)*cos(phi): 중력 수직 성분 (이미 적용 중)
- fmax(..., 0): 차 뜨는 것 방지 (적용 가능)

**수정 체인:**
- `calc_ax_poss`: `dmu_ds` 파라미터 추가, g_tilde 계산
- `__solver_fb_acc_profile`: `dmu_ds` 배열 전달 (backward 시 flipud)
- `__solver_fb_closed`: `dmu_ds` 배열 triple 처리
- `calc_vel_profile`: `dmu_ds` 받아서 전달
- `trajectory_optimizer.py`: create_raceline_3d에서 dmu_ds 추출, calc_vel_profile에 전달
- `racecar_f110.ini`에서 `cog_z` 읽기 (centrifugal 항 옵션용)

### 3-3. 2D ggv 생성 스크립트

`stack_master/scripts/generate_ggv.py` 생성:

**입력** (기존과 동일한 튜닝 파라미터):
```python
v_max = 12.0
ax_max_tire = 7.0      # 타이어 종방향 한계 (튜닝)
ay_max_tire = 5.5       # 타이어 횡방향 한계 (튜닝)
ax_max_motor = 7.0      # 모터 가속 한계
ax_max_brake = 7.0      # 브레이크 감속 한계
```

**출력** (CSV):
- `veh_dyn_info/ggv.csv`: `[v, ax_max, ay_max]`
- `veh_dyn_info/ax_max_machines.csv`: `[v, ax_max_motor]`
- `veh_dyn_info/b_ax_max_machines.csv`: `[v, ax_max_brake]`

**F1TENTH 특성:** 모터 파워(~1100W) >> 타이어 한계 전 속도 구간.
`P_max/(m*v) = 1100/(4.38*12) = 20.9 m/s²` >> `ax_max_tire = 7 m/s²`
→ 파워 제한 ggv는 실질적으로 flat. 하지만 올바른 형태를 갖추는 것이 중요.

**파제카 옵션:** `racecar_f110.ini`에 Magic Formula 계수 (B, C, E)가 있음.
실차 타이어 데이터와 검증되면 물리 기반 ggv 자동 생성 가능.
현재는 미검증 — placeholder 취급.

### 3-4. 3D NLP 경로에 우리 속도 프로파일 입히기

3D NLP 옵티마이저가 만든 경로에 vel_planner_25d의 forward-backward 적용:

```
3D NLP raceline CSV (s_opt, v_opt, n_opt, ...)
    |
    v  sn2cartesian → (x, y, z) 궤적
    |
    v  kappa, el_lengths, slope, dmu_ds 계산
    |
    v  calc_vel_profile(kappa, el_lengths, slope, dmu_ds, ggv, ...)
    |
    v  우리 ggv/튜닝 파라미터로 새 속도 프로파일
```

**장점:**
- 3D NLP 경로는 slope를 고려한 경로 선택 (오르막/내리막 전략)
- 우리 속도 프로파일은 독립적으로 튜닝 가능 (ggv, dyn_exp 등)
- **양쪽의 장점**: 3D 인식 경로 + 강건한 속도 튜닝

**대안:** CasADi NLP로 경로 고정, V(s)만 최적화.
jerk 제한, 타이어 과도 특성 등 더 정밀 가능하나 구현 복잡.
